## 一、实践目标

1. 基于 Open5GS（开源 4G LTE / 5G 移动核心网）+ srsRAN（开源软件无线电套件）+ USRP B210（软件定义无线电平台）+ 带可读写白卡的真实手机，搭建一个端到端的 4G 移动通信网络。
2. 在搭建好的移动核心网之上，集成 Kamailio IMS（P-CSCF / I-CSCF / S-CSCF），提供 VoLTE 多媒体服务。
3. 将实践 1 实现的 PC 客户端、实践 2 实现的 Android 客户端接入上述网络，实现音视频互通。

> 说明：本实践的目标形态是 **USRP B210 + 真实白卡手机走 LTE 空口**。在等待 USRP 到位期间，我们先用 srsRAN 自带的 srsENB / srsUE **ZMQ 虚拟射频**方式，把「核心网 → 接入网 → IMS → VoLTE 呼叫」这条链路做**硬件到货前的过渡验证**，以证明除射频空口外的全部环节均已打通，从而在接入 USRP 后把现场调试范围收敛到射频/空口一段。最终演示与结论以 USRP 真机为准。

## 二、系统总体架构

目标形态的整体拓扑如下（空口由 USRP B210 承载；括注部分为硬件到货前的 ZMQ 过渡验证路径，二者除空口实现外完全一致）：

```text

 │  —— 目标形态 ——   │          │  （非最终演示形态）        │
 └────────┬─────────┘          └────────┬─────────────────┘
          │  LTE 空口                     │  ZMQ 虚拟射频（代替射频硬件）
     ┌────▼─────┐                   ┌────▼─────┐
     │ USRP B210 │                   │  (纯软件) │
     └────┬─────┘                   └────┬─────┘
          └───────────┬─────────────────┘
                 ┌────▼─────────┐
                 │ srsRAN 4G eNB │  PLMN 001/01, TAC 1
                 └────┬─────────┘
                      │ S1AP (SCTP 36412) / GTP-U

        │  MME ─ HSS ─ SGW-C/U ─ PGW(SMF/UPF) ─ PCRF  │
        └───────┬──────────────────────┬──────────────┘

        ┌───────▼──────────────────────▼──────────────┐
        │        Kamailio IMS (Docker 容器)            │
```

## 四、实施步骤

### 4.1 核心网部署（Open5GS EPC）

做了什么：

- 拉取并配置上游 `third_party/docker_open5gs`，编写 `.env.shijian` 统一管理 PLMN（001/01）、TAC（1）、宿主机 IP、UE 子网（internet APN `192.168.100.0/24`、ims APN `192.168.101.0/24`）与射频默认参数。
- 编写 `scripts/build-images.sh`（支持 pull/build 两种模式）准备镜像。
- 编写 `scripts/start-core-ims.ps1` / `stop-core-ims.ps1` 一键启停整套 4G VoLTE 编排（21 个容器）。
- 通过 `config/docker-compose.enb-external.override.yaml` 暴露 S1AP 和 GTP-U 端口，支持外部 eNB 接入。

遇到的问题与解决：

- **Docker Desktop 的 Ubuntu WSL daemon bridge 异常**：Docker Desktop 对 Ubuntu 发行版的 WSL 集成代理一直连不上（属 Docker Desktop 集成代理状态问题，并非 Ubuntu 缺包），WSL 里的 `docker` 命令无法使用。解决办法是绕开 WSL 集成，**核心网改走 Windows 侧 Docker**：编写了对应的 PowerShell 启停/开户脚本（`start-core-ims.ps1`、`provision-subscribers.ps1`、`provision-pyhss.ps1` 等），当 Docker Desktop 弹出 WSL integration 超时窗口时选择 "Skip WSL distro integration"，再用 `-SkipConfigure` 参数以已生成的配置直接启动。核心网运行不受影响。

### 4.2 IMS 部署（Kamailio P/I/S-CSCF + rtpengine）

做了什么：

- 使用 `4g-volte-deploy.yaml` 一并拉起 Kamailio 三个 CSCF、pyHSS、DNS、rtpengine。
- P-CSCF 位于 `172.22.0.21:5060`（UDP/TCP），Server 头标识为 `TelcoSuite Proxy-CSCF`；rtpengine 位于 `172.22.0.16`，负责媒体锚定/中继。
- 验证了三个 CSCF 与 pyHSS 之间的 Cx Diameter 连接（端口 3875）均已建立。

遇到的问题与解决：

- **P-CSCF 的 WITH_RX 门禁导致软电话 403**：P-CSCF 默认开启 `#!define WITH_RX`（`pcscf/pcscf.cfg` 第 140 行附近），REGISTER 时会经 Rx 接口向 PCRF 校验该 UE 的 IP 是否存在 Gx 承载会话。真机/仿真 UE 走 LTE 的 `ims`/`internet` 承载时有 Gx 会话，可以通过；而 PC 端软电话没有蜂窝承载、来自任意 IP，会被直接拒绝 `403 Can't register to QoS for signalling`。解决办法：将 `#!define WITH_RX` 注释为 `##!define WITH_RX` 后重启 pcscf 容器（原配置已备份为 `pcscf/pcscf.cfg.with_rx.bak`）。目前联调环境已放开该门禁；真机走空口注册本不依赖此改动，最终演示是否恢复"真机专用承载 QoS"由小组决定。

### 4.4 用户开户（Open5GS HSS + pyHSS）

做了什么：

- 以 `config/subscribers.csv` 为单一数据源，编写 `scripts/provision-subscribers.ps1/.sh` 把卡5、卡6 写入 Open5GS HSS（含 `internet` 与 `ims` 两个 APN 的会话配置）。
- 编写 `scripts/provision-pyhss.ps1` 向 pyHSS 写入：APN `internet`（apn_id=1）、APN `ims`（apn_id=2），以及两张卡的 AuC（Ki/OPc）、subscriber、IMS subscriber 记录。

关键机制（两种鉴权路径）：

- **真机（白卡）走 LTE 空口**：手机原生 IMS 栈使用 **IMS-AKA**（基于卡内 Ki/OPc），pyHSS 的 AuC 已配好，无需任何改动，这是 2 台真机 VoLTE 的主线。
- **自研软电话（无 SIM）**：Kamailio S-CSCF 的 REGISTER 路由对非 IMS 软电话做了兼容（配置注释 `force to MD5 for zoiper.... non-ims`），当 Authorization 未携带 AKA 算法时强制走 **MD5 SIP Digest**；pyHSS 处理 `Digest-MD5` 时直接把该卡的 **Ki 当作口令**返回给 S-CSCF 校验（`lib/database.py` 中 `vector_dict['SIP_Authenticate'] = key_data['ki']`）。
- pyHSS 用 IMPI 中 `@` 之前的部分作为 IMSI 查订户，因此软电话的 IMPI/IMPU 用户名部分必须填 IMSI；互相呼叫时被叫号即对方 IMSI。

软电话接入 IMS 的配置配方（已实测注册成功）：

| 项 | 值 |
| --- | --- |
| 注册地址/AOR（IMPU） | `sip:001012345678905@ims.mnc001.mcc001.3gppnetwork.org` |
| 鉴权用户名（IMPI） | `001012345678905@ims.mnc001.mcc001.3gppnetwork.org` |
| 口令 | 该卡 Ki：`000102030405060708090A0C0B0D0E0F` |
| Realm/域 | `ims.mnc001.mcc001.3gppnetwork.org` |
| 注册/代理服务器 | P-CSCF `172.22.0.21:5060`（真机走 LTE 时由核心网下发/路由） |
| 传输 | UDP，Contact 必须带 `;transport=udp` |

对自研 `SipClient`（实践 1/2 客户端）：登录"用户名"填 IMSI、"密码"填 Ki、"服务器"填 IMS 域名即可，无需修改客户端代码（其 URI 按 `user@server` 拼装、realm 从 401 挑战中获取、已支持 qop=auth 的 MD5 Digest）。

### 4.6 客户端接入（实践 1 PC 端 + 实践 2 Android 端）

做了什么：

- 按 4.4 节配方配置实践 1 PC 客户端与实践 2 Android 客户端的 SIP 账号（IMSI 作用户名、Ki 作口令、IMS 域作 realm/服务器）。
- 编写并运行验证脚本：`scripts/ims_register_test.py`（单端 REGISTER 验证）、`scripts/ims_call_test.py`（卡5/卡6 双端注册 + 主叫 INVITE 验证）。脚本需在 `docker_open5gs_default` 容器网络内运行（宿主机无法直达 172.22.0.x 网段），方法是 `docker cp` 进 pyhss 容器后 `docker exec pyhss python3 /tmp/xxx.py`。

遇到的问题与结论：

- **终呼 477 的真正根因（已定位并修复）**：早期复测终呼时出现 `477 error on sending to next hop`。深入排查确认其**根因不是端点部署形态**，而是 P-CSCF 配置 `kamailio_pcscf.cfg` 中的 `udp_mtu = 1300` + `udp_mtu_try_proto = TCP`——任何大于 1300 字节的终呼 INVITE（带完整音视频 SDP 时即超过）会被核心**静默改用 TCP** 投递到被叫；若被叫软电话只监听 UDP，TCP 连接被拒（`Connection refused`），P-CSCF 遂回 477。解决办法：注释该两行并重启 pcscf（原文件备份 `kamailio_pcscf.cfg.udp_mtu.bak`）；或让客户端在同端口同时监听 TCP。修复后终呼稳定走 UDP。
- **对话内 ACK/BYE 得到 404 Not here**：本栈 P-CSCF 对 UDP 客户端的 200 OK 不回 Record-Route，UAC 路由集为空，按 RFC 3261 应将 ACK/BYE 直接发往对端 Contact，而非发回 P-CSCF。据此修正测试端点后 ACK/BYE 均达对端。
- **端到端呼叫已闭环**：以两个独立 IP 端点（主叫在 ZMQ 的 srsUE 172.22.0.34、被叫在独立容器 172.22.0.26）实测完整链路 **REGISTER 200 → INVITE → 100/180 → 200 OK（rtpengine 锚定 audio+video）→ ACK → BYE 200 OK**，主被叫双方日志均确认收发一致，通话干净建立与挂断。

## 七、总结与展望

本次实践在 Windows 11 + WSL2 + Docker 的单机环境中完成了 Open5GS EPC 与 Kamailio IMS（含 pyHSS、DNS、rtpengine）共 21 个网元的容器化部署，打通了 HSS/pyHSS 双侧开户流程，并将自研 SIP 客户端经 MD5 Digest 成功注册进 IMS，实测端到端 VoLTE 呼叫（REGISTER → INVITE → 200 OK → ACK → BYE）完整穿过全部 CSCF 并触发 rtpengine 完成音视频媒体锚定，验证了"核心网—接入网—IMS—客户端"整条链路的正确性。过程中定位并解决了 Docker Desktop WSL 集成异常（改走 Windows Docker）、P-CSCF WITH_RX 门禁 403（按需放开）、终呼 477（根因为 `udp_mtu` 强制切 TCP，已修复）、对话内 404（路由集为空时直发 Contact）等具体工程问题，并在硬件到货前用 ZMQ 虚拟射频完成了 srsUE 全软件 attach 的过渡验证。
